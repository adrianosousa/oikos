# OIKOS P2P MARKETPLACE — Oikos Policy Engine Skill

## IDENTITY

- Module: Oikos-native (Hyperswarm DHT + Protomux channels)
- Actions: PUBLISH_LISTING, REMOVE_LISTING, BROWSE_LISTINGS, INITIATE_TRADE, ACCEPT_TRADE, COMMIT_PAYMENT, CONFIRM_RECEIPT, COMPLETE_TRADE, DISPUTE_TRADE, CANCEL_TRADE
- Confirmation Tier: BROWSE_LISTINGS = Tier 0, all others = Tier 1 or 2

## WHAT IT DOES

A fully decentralized peer-to-peer marketplace where sovereign agents discover each other via Hyperswarm DHT, negotiate trades over Protomux encrypted channels, and settle payments using the best available WDK payment rail for the trade object. The protocol is a deterministic state machine — no LLM judgment is involved in settlement sequencing.

## WHAT IT DOES NOT DO

- Does NOT provide escrow (no trusted third party exists in P2P)
- Does NOT guarantee counterparty honesty (reputation + small increments mitigate this)
- Does NOT enforce delivery of off-chain goods/services (only on-chain payment can be verified)
- Does NOT support fiat settlement (use MoonPay for fiat on/off-ramp, then trade in crypto)
- Does NOT perform price discovery (prices are set by listing publishers)

## 1. LISTING SCHEMA

Every listing announced on the DHT follows this exact schema. No optional cleverness — every field is present or the listing is invalid.

```json
{
  "listing_id": "oikos_ls_<uuid>",
  "role": "SELLER",
  "agent_id": "oikos_ag_<public_key_short>",
  "created_at": 1710672000,
  "ttl_seconds": 3600,
  "description": "Offering 0.01 BTC for USDT0 at market rate",
  "offer": {
    "asset": "BTC",
    "chain": "bitcoin",
    "amount": "1000000",
    "amount_display": "0.01 BTC"
  },
  "price": {
    "amount": "850.00",
    "denomination": "USDT0",
    "denomination_chain": "arbitrum",
    "type": "FIXED"
  },
  "accepted_rails": ["spark", "bitcoin", "evm:arbitrum", "evm:ethereum"],
  "metadata": {
    "min_trade_amount": "100000",
    "max_trade_amount": "1000000",
    "min_trade_display": "0.001 BTC",
    "max_trade_display": "0.01 BTC"
  }
}
```

### Field Definitions

| Field | Description |
|-------|-------------|
| `listing_id` | Unique identifier. Generated by the engine, not the LLM. |
| `role` | `"BUYER"` or `"SELLER"` — the publisher's role in the trade. SELLER: "I have X and want Y in return". BUYER: "I want X and will pay Y". |
| `agent_id` | The publisher's public identity (derived from Hyperswarm keypair). |
| `created_at` | Unix timestamp. Set by the engine. |
| `ttl_seconds` | How long the listing stays active on DHT. Engine enforces expiry. |
| `description` | Free-text description. The LLM writes this. It is DISPLAY ONLY — the engine uses structured fields for all logic. |
| `offer.asset` | Human-readable asset name (for display). Engine resolves via token registry. |
| `offer.chain` | WDK chain key where the asset lives. |
| `offer.amount` | Amount in smallest unit (string). |
| `offer.amount_display` | Human-readable amount (for display). |
| `price.amount` | Price in the denomination asset. |
| `price.denomination` | The asset they want to receive. |
| `price.denomination_chain` | Which chain the denomination asset is on. |
| `price.type` | `"FIXED"` or `"MARKET"` (market = re-quoted at trade initiation). |
| `accepted_rails` | Which payment methods the publisher will accept. Format: `"spark"`, `"bitcoin"`, `"evm:<chain_key>"`. |
| `metadata` | Trade size limits. |

### Listing Validation Rules (engine-enforced)

- `role` must be `"BUYER"` or `"SELLER"` — nothing else
- `offer.chain` must be registered in the publisher's wallet
- For SELLER listings: engine verifies balance >= `offer.amount` at publish time
- `ttl_seconds` must be in [60, 86400] (1 minute to 24 hours)
- `accepted_rails` must only contain rails the publisher's wallet actually supports
- `listing_id` is generated by engine (agent cannot pick its own)
- `created_at` is set by engine (agent cannot backdate)
- `description` is max 280 characters (prevent prompt injection via long text)
- An agent can have at most 5 active listings (prevent DHT spam)

## 2. THE TRADE STATE MACHINE

This is the core of the settlement protocol. Every trade progresses through these states in order. There are no shortcuts, no states the LLM can skip, and no branching that requires judgment.

```
┌──────────┐
│ PROPOSED │ ← Buyer sends INITIATE_TRADE
└────┬─────┘
     │ Seller sends ACCEPT_TRADE
     ▼
┌──────────┐
│ ACCEPTED │ ← Both parties have agreed on terms
└────┬─────┘
     │ Buyer sends COMMIT_PAYMENT
     │ (includes tx_hash or payment proof)
     ▼
┌──────────────┐
│ PAYMENT_SENT │ ← Buyer claims to have paid
└────┬─────────┘
     │ Seller verifies payment on-chain/Lightning
     │ Seller sends CONFIRM_RECEIPT
     ▼
┌────────────────────┐
│ PAYMENT_CONFIRMED  │ ← Seller acknowledges payment received
└────┬───────────────┘
     │ Seller sends COMPLETE_TRADE
     │ (delivers asset: sends BTC, RGB consignment, etc.)
     ▼
┌───────────┐
│ COMPLETED │ ← Trade is done. Both stash trade record.
└───────────┘
```

At any state before COMPLETED:
- Either party can send `CANCEL_TRADE` → state moves to CANCELLED
- Either party can send `DISPUTE_TRADE` → state moves to DISPUTED

### Why This Order Is Correct

**The buyer pays first.** This is a deliberate design choice:

1. On-chain payments are VERIFIABLE. The seller can independently confirm the payment arrived by checking the blockchain or Lightning preimage. The seller doesn't have to trust the buyer's claim — they verify.

2. Off-chain deliveries are NOT universally verifiable. If the seller sends a service, file, or off-chain asset, the buyer could claim they never received it. There's no on-chain proof of delivery for arbitrary goods.

3. Therefore: the verifiable leg goes first (payment), the trust-dependent leg goes second (delivery). This minimizes the attack surface.

4. The remaining risk: seller takes payment and doesn't deliver. Mitigation: reputation system + incremental trades + dispute mechanism. For high-value trades, use multiple smaller increments.

### State Transition Rules (deterministic, engine-enforced)

**PROPOSED → ACCEPTED:**
- Only the SELLER (listing publisher) can accept.
- Timeout: if seller doesn't accept within ACCEPT_TIMEOUT (default: 300s), → EXPIRED.
- Engine checks: seller still has sufficient balance for the offered asset.

**ACCEPTED → PAYMENT_SENT:**
- Only the BUYER can transition.
- Buyer must provide: `{ tx_hash, rail_used, amount_sent }`
- Engine validates:
  - `rail_used` is in the listing's `accepted_rails`
  - `tx_hash` format matches the rail (e.g., 64-char hex for BTC, 0x... for EVM)
- Timeout: if buyer doesn't pay within PAYMENT_TIMEOUT (default: 600s), → EXPIRED.

**PAYMENT_SENT → PAYMENT_CONFIRMED:**
- Only the SELLER can transition.
- Engine assists verification:
  - For Spark/Lightning: check preimage is valid for the payment hash
  - For Bitcoin L1: check tx confirmations >= REQUIRED_CONFIRMATIONS (default: 1)
  - For EVM: check tx receipt status == success and amount matches
- If verification fails: seller can DISPUTE_TRADE instead of confirming.
- Timeout: if seller doesn't confirm within CONFIRM_TIMEOUT (default: 1800s), → DISPUTED.

**PAYMENT_CONFIRMED → COMPLETED:**
- Only the SELLER can transition.
- Seller delivers the trade object:
  - For crypto-for-crypto: seller sends the offered asset via appropriate module (SEND_TRANSACTION, RGB_TRANSFER, etc.)
  - For services: seller marks delivery complete (inherently trust-based)
- Seller provides: `{ delivery_tx_hash (if applicable), delivery_proof }`
- Engine checks: if delivery is an on-chain send, verifies the tx was broadcast.

**CANCEL_TRADE:**
- Either party, at any state before COMPLETED.
- If state == PAYMENT_SENT or PAYMENT_CONFIRMED: payment was already sent. The engine WARNS: "Payment has already been sent. Cancellation does not reverse it."
- Cancellation is a RECORD, not an undo mechanism.

**DISPUTE_TRADE:**
- Either party, at any state before COMPLETED.
- Dispute is recorded with `{ reason, evidence_hash, timestamp }`.
- In V1: disputes are logged for reputation purposes. No automated resolution.
- Future: could integrate with a decentralized arbitration protocol.

## 3. PAYMENT RAIL SELECTION

The agent picks the settlement rail based on the trade object and the listing's `accepted_rails`. This is the ONE place where the LLM makes a constrained choice.

### Rail Selection Rules (deterministic priority)

**For BTC trades:**
1. If both parties support Spark → use Spark (instant, lowest fees)
2. If Spark unavailable but Lightning invoice provided → use Lightning
3. Fallback: Bitcoin L1 (slow, higher fees, but always works)

**For USDT/stablecoin trades:**
1. If both parties have the same EVM chain → direct EVM send (cheapest gas)
2. If different EVM chains → bridge via USDT0, then send
3. If seller accepts Spark and USDT is available on Spark → use Spark

**For RGB asset trades:**
1. RGB_TRANSFER via consignment over Hyperswarm (the only option)
2. Settlement of the payment leg follows the rules above

The engine presents the selection to the LLM as a ranked list. The LLM picks from the list. The LLM CANNOT invent a rail not in the list.

### Rail Selection Schema

Engine provides to LLM:

```json
{
  "available_rails": [
    { "rail": "spark", "fee_estimate": "2 sats", "speed": "instant", "recommended": true },
    { "rail": "evm:arbitrum", "fee_estimate": "0.0001 ETH", "speed": "~15 seconds", "recommended": false },
    { "rail": "bitcoin", "fee_estimate": "0.00005 BTC", "speed": "~10 minutes", "recommended": false }
  ]
}
```

LLM picks: `{ "selected_rail": "spark" }`

Engine validates: `"spark"` is in listing's `accepted_rails` → proceed.

## 5. DETERMINISTIC FLOW — FULL TRADE LIFECYCLE (BTC for USDT0 example)

```
═══════════════════════════════════════════════════
SELLER (Agent A): Has 0.01 BTC, wants USDT0
BUYER  (Agent B): Has USDT0, wants BTC
═══════════════════════════════════════════════════
```

1. **SELLER publishes listing on DHT**
   - Engine: Validates balance (0.01 BTC ≥ offer amount), announces via Hyperswarm
   - DHT now contains: { role: SELLER, offer: 0.01 BTC, price: 850 USDT0, rails: [spark, evm:arbitrum] }

2. **BUYER browses listings, finds SELLER's listing**
   - Engine: Returns listing data. LLM presents to human.
   - Human: "Buy 0.005 BTC from this listing"

3. **BUYER initiates trade**
   - Engine: Creates trade_id, opens Protomux channel to SELLER
   - Sends INITIATE_TRADE: { listing_id, trade_amount: 500000, selected_rail: spark }
   - Trade state → PROPOSED

4. **SELLER's engine receives proposal**
   - Engine: Checks balance still sufficient, checks trade_amount within limits
   - LLM: "Agent B wants to buy 0.005 BTC for 425 USDT0 via Spark. Accept?"
   - Seller human: "yes"
   - Engine: Sends ACCEPT_TRADE
   - Trade state → ACCEPTED

5. **BUYER's engine receives acceptance**
   - Engine: Computes payment details:
     - Rail: Spark
     - Amount: 425 USDT0 (or equivalent in sats if paying in BTC via Spark)
     - SELLER provides Spark invoice or address
   - Engine: Presents rail options to LLM → LLM confirms Spark
   - Engine: Executes payment via wallet-spark (PAY_LIGHTNING_INVOICE or Spark transfer)
   - Engine: Gets tx_hash/preimage
   - Engine: Sends COMMIT_PAYMENT with proof
   - Trade state → PAYMENT_SENT

6. **SELLER's engine receives payment proof**
   - Engine: Verifies payment:
     - Spark: validates preimage against payment hash
     - EVM: checks tx receipt on-chain
     - BTC L1: checks tx in mempool/confirmed
   - If valid:
     - LLM: "Payment of 425 USDT0 received via Spark. Confirmed."
     - Engine: Sends CONFIRM_RECEIPT
     - Trade state → PAYMENT_CONFIRMED
   - If invalid:
     - Engine: Sends DISPUTE_TRADE with reason

7. **SELLER delivers the trade object**
   - Engine: Executes SEND_TRANSACTION via wallet-btc (send 0.005 BTC to buyer)
   - Engine: Gets delivery_tx_hash
   - Engine: Sends COMPLETE_TRADE with delivery proof
   - Trade state → COMPLETED

8. **BUYER's engine verifies delivery**
   - Engine: Checks BTC arrived (via wallet-btc balance update)
   - LLM: "Trade completed. Received 0.005 BTC. Tx: btc_abc..."
   - Both engines archive trade record with full audit trail.

## 6. TIMEOUT AND FAILURE HANDLING

**ACCEPT_TIMEOUT (default: 300s / 5 min):**
- If seller doesn't accept → trade EXPIRED
- Buyer's engine: "Trade proposal expired. Seller did not respond."
- No funds moved. Clean exit.

**PAYMENT_TIMEOUT (default: 600s / 10 min):**
- If buyer doesn't pay after acceptance → trade EXPIRED
- Seller's engine: listing remains active for other buyers.
- No funds moved. Clean exit.

**CONFIRM_TIMEOUT (default: 1800s / 30 min):**
- If seller doesn't confirm receipt after buyer pays → trade DISPUTED
- This is the danger zone: buyer has paid, seller hasn't acknowledged.
- Engine: auto-creates dispute record with payment proof.
- LLM to buyer: "Seller has not confirmed your payment after 30 minutes. A dispute has been recorded. Your payment proof: {tx_hash}."

**DELIVERY_TIMEOUT (default: 3600s / 1 hour):**
- If seller confirms receipt but doesn't deliver → trade DISPUTED
- Engine: auto-creates dispute with { payment_confirmed, delivery_missing }.

**RECONNECTION:**
- If Protomux connection drops mid-trade:
  - Trade state is persisted locally by both engines
  - On reconnect: engines exchange current trade state and resume
  - If states conflict: the more advanced state wins (state machine is monotonic)
  - If peer never reconnects: trade moves to DISPUTED after relevant timeout

## 7. GUARDRAILS (P2P-SPECIFIC)

**GUARDRAIL_11: Self-Trade Prevention**
- Agent CANNOT initiate a trade on its own listing.
- Agent CANNOT pay itself.
- Engine compares counterparty `agent_id` against own `agent_id`.

**GUARDRAIL_P2P_01: Listing Balance Lock**
- When a SELLER publishes a listing, the offered amount is "soft-locked": the engine warns if subsequent operations would reduce balance below the listed amount.
- Not a hard lock (funds aren't frozen) but a guardrail against accidental spending.

**GUARDRAIL_P2P_02: Concurrent Trade Limit**
- Max concurrent active trades: configurable (default: 3).
- Prevents the agent from overcommitting.

**GUARDRAIL_P2P_03: Counterparty Message Sanitization**
- All messages from peers (descriptions, reasons, etc.) are treated as UNTRUSTED.
- They are NEVER fed to the LLM as instructions.
- They are presented to the human with a prefix: "Peer says: {message}"
- Max message length: 280 characters. Truncated if longer.

**GUARDRAIL_P2P_04: Price Sanity Check**
- If listing price deviates more than PRICE_DEVIATION_THRESHOLD (default: 20%) from a reference rate (if available via Price Rates tool), warn the human:
  "This listing's price is {deviation}% {above/below} market rate."
- This is a WARNING, not a block — the human decides.

**GUARDRAIL_P2P_05: Payment Verification Before Confirmation**
- The seller's engine MUST independently verify payment on-chain/Lightning before allowing CONFIRM_RECEIPT.
- The LLM CANNOT confirm receipt based on the buyer's claim alone.
- The engine checks the actual chain/Lightning state.

## 11. INTERACTION WITH OTHER MODULES

**wallet-btc (01):**
- Seller delivers BTC via SEND_TRANSACTION when trade object is Bitcoin.
- Engine coordinates: trade COMPLETE_TRADE triggers the BTC send.

**wallet-evm (02):**
- Payment rail for USDT0/stablecoin settlement on EVM chains.
- Also used when trade object is an EVM token.

**wallet-spark (05):**
- PRIMARY settlement rail for instant trades.
- Seller creates invoice → buyer pays → preimage serves as payment proof.
- Fastest path: trade settlement in < 1 second.

**swap-velora (10):**
- If buyer has the wrong asset, engine can suggest:
  "You have ETH but this listing wants USDT0. Swap first?"
- The swap happens BEFORE the trade, not during.

**bridge-usdt0 (11):**
- If buyer's USDT0 is on wrong chain, engine can suggest bridging first.
- Bridge happens BEFORE the trade.

**lending-aave (12):**
- If buyer has assets supplied to Aave, engine can suggest withdrawal first.
  "Your USDT0 is earning yield on Aave. Withdraw to fund this trade?"

**fiat-moonpay (13):**
- If buyer has no crypto at all, engine suggests on-ramping first.
  "Insufficient balance. Buy USDT0 via MoonPay?"

**wallet-rgb (15):**
- For trades where the object is an RGB token:
  - Settlement of payment leg uses standard rails (Spark, EVM, etc.)
  - Delivery of RGB asset uses RGB_TRANSFER with consignment over Hyperswarm
  - Both legs are coordinated by the trade state machine
